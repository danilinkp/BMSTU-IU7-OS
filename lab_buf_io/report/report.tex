\subsection*{Первая программа. Один поток}
\begin{lstlisting}[language=C]
#include <stdio.h>
#include <fcntl.h>

int main()
{
	int fd = open("alphabet.txt", O_RDONLY); 
	FILE *fs1 = fdopen(fd, "r");
	char buff1[20];
	setvbuf(fs1, buff1, _IOFBF, 20);
	FILE *fs2 = fdopen(fd, "r");
	char buff2[20];
	setvbuf(fs2, buff2, _IOFBF, 20);
	int flag1 = 1, flag2 = 2;
	while (flag1 == 1 || flag2 == 1)
	{
		char c;
		flag1 = fscanf(fs1, "%c", &c);
		if (flag1 == 1)
		{
			fprintf(stdout, "%c", c);
		}
		flag2 = fscanf(fs2, "%c", &c);
		if (flag2 == 1)
		{
			fprintf(stdout, "%c", c);
		}
	}
	printf("\n");
	return 0;
}
\end{lstlisting}
Данная программа считывает информацию из файла <<alphabet.txt>>, который содержит строку символов <<abcdefghigklmnopqrstuvwxyz>>.
И при помощи двух буферов посимвольно выводит считанные символы в стандартный поток вывода stdout.

В начале функции main open() создает новый файловый дескриптор для открытого
только на чтение (O\_RDONLY) файла <<alphabet.txt>>, запись в системной таблице открытых файлов. 
Эта запись регистрирует смещение в файле и флаги состояния файла.

Далее fdopen() создает два указателя на структуру FILE, приведенную выше.
В данных структурах поле \_fileno будет содержать дескриптор, который вернула
функция fopen(). Для fs1 и fs2 эти поля будут равны 3.

Функция setvbuf() изменяет тип буферизации для fs1 и fs2 на
полную буферизацию, а также явно задает размер буфера 20 байт.

Далее при первом вызове fscanf() буфер fs1 заполнится полностью, 
т.е. первыми 20 символами. 
Значение f\_pos в структуре struct\_file открытого файла увеличится на 20.
Далее при первом вызове fscanf() для fs2 в buff2 считаются оставшиеся 6 символов,
начиная с f\_pos (т.к. fs1 и fs2 ссылаются на один и тот же дескриптор fd).

Далее в цикле поочередно выводятся символы из buff1 и buff2.
Т.к. в buff2 записались оставшиеся 6 символов, после 6 итерации
цикла будут выводится символы только из buff1.

\subsection*{Вторая программа. Один поток}
\begin{lstlisting}[language=C]
#include <fcntl.h>
#include <unistd.h> // read, write.

int main()
{
	char c;
	int fd1 = open("alphabet.txt", O_RDONLY);
	int fd2 = open("alphabet.txt", O_RDONLY);
	int rc1, rc2 = 1;
	while (rc1 == 1 || rc2 == 1){
		char c;
		rc1 = read(fd1, &c, 1);
		if (rc1 == 1){
			write(1, &c, 1);}
		rc2 = read(fd2, &c, 1);
		if (rc2 == 1){
			write(1, &c, 1);}
	}
	write(1, "\n", 1);
	return 0;
}
\end{lstlisting}

\subsection*{Вторая программа. Два потока}
\begin{lstlisting}[language=C]
#include <fcntl.h>
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

void *reader(void *arg)
{
    int fd = open("alphabet.txt", O_RDONLY);
    char c;
    int flag = 1;
    while (flag == 1)
    {
        flag = read(fd, &c, 1);
        if (flag == 1)
        {
            write(1, &c, 1);
        }
    }
    close(fd);
    pthread_exit(NULL);
}

int main(void)
{
    pthread_t thread1, thread2;
    pthread_create(&thread1, NULL, reader, NULL);
    pthread_create(&thread2, NULL, reader, NULL);
    pthread_join(thread1, NULL);
    pthread_join(thread2, NULL);
    write(1, "\n", 1);
    return 0;
}
\end{lstlisting}

В программе также, как и при реализации с одним потоком, создается
два файловых дескриптора для открытого файла, записи в системной таблице открытых файлов.
У каждой записи будет свое смещение f\_pos.
Т.к. главный поток ждет окончания дочернего, то гарантируется вывод
всего алфавита дважды.
В многопоточной реализации каждый символ будет выведен дважды, но
последовательность символов не определена.

\textbf{Буферизованный ввод.}
При вызове \texttt{fopen()} дважды открывается файл \texttt{3.txt}.
Создаются два файловых дескриптора и две структуры \texttt{struct FILE},
поля \texttt{\_fileno} которых инициализируются соответствующими значениями дескрипторов.
 
В цикле с помощью системного вызова \texttt{(write())} производится запись чётных буквы алфавита в буфер первого дескриптора, нечётных~--- в буфер второго. Запись в файл из буфера
происходит в одном из трёх случаев:
\begin{itemize}
  \item буфер заполнен;
  \item вызвана функция \texttt{fflush()};
  \item вызвана функция \texttt{fclose()}.
\end{itemize}
 
При вызове \texttt{fclose()} для первой структуры в файл из буфера записываются нечетные буквы латинского алфавита.
Так как поле \texttt{f\_pos} структуры \texttt{struct file}, соответствующей второму
дескриптору, не изменилось, при последующем вызове \texttt{fclose()} чётные буквы
записываются на те же позиции, перезаписывая ранее сохранённые данные.
Вывод полей структуры \texttt{stat} подтверждает этот результат: размер файла
не изменяется в ходе цикла, поскольку до закрытия дескриптора физической
записи не происходит.
 
\bigskip
 
\textbf{Небуферизованный ввод}
Системные вызовы \texttt{open()} открывают файл \texttt{3.txt} для чтения и записи,
создавая дескрипторы со значениями~3 и~4 (значения 0,~1,~2 заняты стандартными
потоками \texttt{stdin}, \texttt{stdout}, \texttt{stderr}), соответствующие
индексам в таблице дескрипторов файлов процесса.
 
Поля \texttt{fd\_array[3]} и \texttt{fd\_array[4]} указывают на разные структуры
\texttt{struct file}, связанные с одной и той же \texttt{struct inode}.
Поскольку созданы две независимые структуры, их поля \texttt{f\_pos} изменяются
независимо. В результате чётные и нечётные буквы латинского
алфавита записываются поочерёдно на одни и те же позиции файла, а размер файла
не изменяется, что подтверждается выводом \texttt{stat()} в ходе выполнения программы.

\subsection*{Вывод}

При работе с буферезированным вводом-выводом может возникнуть ряд
проблем, которые демонстритруются в программах данной лабораторной работы.
В первой программе использование stdio, который использует буферизованный ввод/вывод, приводит к тому, что данные сначала записываются в буфер, а затем выбираются из буфера.
Во второй программе возникает проблема вывода символов на
экран, так как из-за двух файловых дескрипторов смещения в файле будут
производиться независимо. 
В третьей программе при одновременной записи данные могут быть перезаписаны, по этой причине необходимо открывать файл с опцией "a" (O\_APPEND), тогда операция записи станет атомарной и данные не будут потеряны.
